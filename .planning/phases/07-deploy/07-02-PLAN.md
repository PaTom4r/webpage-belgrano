  <what>
Claude deployed the site to Vercel. Now you need to:
1. Set the 3 environment variables so the contact form sends real emails
2. Connect belgrano.cl to the deployment
3. Set up DNS at your domain registrar

**Step 1 — Add environment variables in Vercel:**

Go to: https://vercel.com/patom4r/webpage-belgrano/settings/environment-variables

Add these 3 variables (all environments: Production, Preview, Development):

| Variable | Value |
|----------|-------|
| `RESEND_API_KEY` | Your API key from resend.com → API Keys |
| `RESEND_FROM_EMAIL` | e.g. `contacto@belgrano.cl` (must be verified in Resend) |
| `TEAM_EMAIL` | The inbox where you want lead notifications, e.g. `contacto@belgrano.cl` |

To get RESEND_API_KEY: log in to resend.com → API Keys → Create API Key (Production mode).

**Step 2 — Verify belgrano.cl domain in Resend (if not done already):**

Go to: resend.com → Domains → Add Domain → type `belgrano.cl`

Resend shows 3 DNS records (TXT + MX + TXT). Add them at your domain registrar DNS panel. Once verified, Resend will let you send from `@belgrano.cl` addresses.

**Step 3 — Connect belgrano.cl in Vercel:**

Go to: https://vercel.com/patom4r/webpage-belgrano/settings/domains

Click "Add Domain" → type `belgrano.cl` → click Add.

Vercel shows the DNS records to add:
- For apex domain: A record → 76.76.21.21
- For www: CNAME → cname.vercel-dns.com

Add these at your domain registrar. DNS can take 5-30 minutes to propagate.

**Step 4 — Trigger a redeploy to pick up the new env vars:**

In Vercel dashboard → Deployments → click the 3 dots on the latest deployment → Redeploy.

Or via CLI: `vercel --prod`
  </what>
  <resume-signal>Type "done" when env vars are set, domain is connected in Vercel, and you have added the DNS records at your registrar. (Domain does not need to be fully propagated yet.)</resume-signal>

  <files>.planning/phases/07-deploy/lighthouse-report.json</files>
  <action>
Run a Lighthouse audit against the production URL. Use the Lighthouse CLI (not Chrome DevTools) so the result is reproducible and saved:

```bash
# Install Lighthouse CLI if not present
npm list -g lighthouse 2>/dev/null || npm install -g lighthouse

# Run mobile audit on homepage (mobile is the key metric per DEPLOY-04)
lighthouse https://belgrano.cl \
  --output=json \
  --output-path=C:/Users/patri/Documents/Code/trabajo/webpage-belgrano/.planning/phases/07-deploy/lighthouse-report.json \
  --form-factor=mobile \
  --screenEmulation.mobile=true \
  --screenEmulation.width=375 \
  --screenEmulation.height=667 \
  --screenEmulation.deviceScaleFactor=2 \
  --chrome-flags="--headless --no-sandbox" \
  --only-categories=performance,accessibility,best-practices,seo
```

If belgrano.cl DNS has not propagated yet, use the Vercel preview URL instead:
```bash
lighthouse https://webpage-belgrano.vercel.app \
  --output=json \
  --output-path=C:/Users/patri/Documents/Code/trabajo/webpage-belgrano/.planning/phases/07-deploy/lighthouse-report.json \
  --form-factor=mobile \
  --screenEmulation.mobile=true \
  --screenEmulation.width=375 \
  --screenEmulation.height=667 \
  --screenEmulation.deviceScaleFactor=2 \
  --chrome-flags="--headless --no-sandbox" \
  --only-categories=performance,accessibility,best-practices,seo
```

Read the JSON report and extract the performance score:
```bash
node -e "const r=require('C:/Users/patri/Documents/Code/trabajo/webpage-belgrano/.planning/phases/07-deploy/lighthouse-report.json'); console.log('Performance:', Math.round(r.categories.performance.score*100), '\nAccessibility:', Math.round(r.categories.accessibility.score*100), '\nSEO:', Math.round(r.categories.seo.score*100))"
```

**If performance score < 90**, apply these targeted optimizations and redeploy:

1. **LCP image not preloaded:** In `src/components/sections/hero-section.tsx`, if using `next/image` for an above-fold image, add `priority` prop. If the hero is text-only (gradient background), check for CLS caused by font loading — verify Geist Sans uses `display: swap` in `layout.tsx` (already configured).

2. **Large JavaScript bundles:** Install `@next/bundle-analyzer` (`npm install --save-dev @next/bundle-analyzer`), then add to `next.config.ts`:
```typescript
import bundleAnalyzer from '@next/bundle-analyzer'
const withBundleAnalyzer = bundleAnalyzer({ enabled: process.env.ANALYZE === 'true' })
export default withBundleAnalyzer(nextConfig)
```
Run `ANALYZE=true npm run build` and open the generated report. Identify components > 50KB not above the fold and wrap them in `dynamic(() => import(...), { ssr: false })`.

3. **Calendly script blocking:** The CalendlyEmbed loads its script via `useEffect` — it should not block initial render. If Lighthouse flags it, verify no static `<script src="calendly.com">` tag exists in the SSR HTML.

4. **Font blocking:** Geist and Inter both use `display: swap` via `next/font/google` — already correct. No action needed unless Lighthouse flags a specific font warning.

After any fixes, rebuild and redeploy:
```bash
cd C:/Users/patri/Documents/Code/trabajo/webpage-belgrano && npm run build && vercel --prod
```

Re-run Lighthouse until score >= 90. Save the final passing report to the path above.
